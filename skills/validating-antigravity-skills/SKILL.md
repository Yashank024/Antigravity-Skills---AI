# Validating Antigravity Skills

This skill acts as the Quality Assurance (QA) pipeline step in the AI Skill Factory. It ensures that any skill generated by `creating-antigravity-skills` (or written manually) perfectly adheres to the Antigravity engine's requirements.

## Trigger Keywords
Activate this skill if the request mentions:
- validate skill
- check skill
- verify skill
- skill validator
- QA skill

## Validation Workflow

When tasked with validating a skill, run the target skill through this strict auditing checklist:

### 1. Folder Structure Check 📂
- [ ] Does the folder exist inside `.agent/skills/` or `.gemini/antigravity/skills/`?
- [ ] Is there exactly one main `SKILL.md` file at the root of the skill folder?
- [ ] If secondary files are present, are they neatly organized into `scripts/`, `examples/`, or `resources/`?

### 2. YAML Frontmatter Check 📝
- [ ] Are both `---` indicators present at the very top of `SKILL.md`?
- [ ] **Name Rule**: Is the `name` field present, under 64 characters, written using lowercase letters and hyphens only, and (ideally) in a gerund format (e.g., ending in `-ing`)?
- [ ] **Name Avoidance**: Does it actively avoid using words like "claude" or "anthropic"?
- [ ] **Description Rule**: Is the `description` field present, under 1024 characters, and written in the third person (e.g., "Generates tests..." instead of "I will generate tests...")?

### 3. Content & Quality Check 🧠
- [ ] **Trigger Keywords**: Is there an explicit section mentioning how or when the agent should trigger the skill?
- [ ] **Progressive Disclosure**: Is the `SKILL.md` concise and strictly under 500 lines? (Are complex details correctly pushed to secondary files?)
- [ ] **Degrees of Freedom**: Does the markdown use bullet points for open-ended heuristics and explicitly formatted code/bash blocks for rigid templates/commands?

## Execution Pattern
Once you perform the audit, generate a **Validation Report** for the user. 
- Use ✅ for perfectly compliant areas.
- Use ❌ for critical failures (like invalid YAML or missing SKILL.md).
- Use ⚠️ for warnings (like description exceeding 1024 chars or missing gerunds).

If failures (`❌`) are found, proactively provide the corrected code snippet that fixes the issue.
